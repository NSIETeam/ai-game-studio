## 协作模式

AI Game Studio 采用 **流水线（Pipeline）+ 扇出（Fan-out）** 混合协作模式。

### 模式一：顺序流水线（Serial Pipeline）

每个 Agent 的输出是下一个 Agent 的输入，形成工作链。

```
  输入创意 → [Designer] → PRD → [Artist] → 视觉设计 → [Architect] → 技术方案
  → [Coder] → 代码 → [Tester] → 审查报告 → [Publisher] → 部署
```

**适用场景**：从零创建新游戏

### 模式二：并行扇出（Parallel Fan-out）

多个 Agent 同时独立工作，互不依赖。

```
  ┌─── [Tester 1] → 审查恐龙跑酷
  │
  输入 →─── [Tester 2] → 审查太空射击
  │
  └─── [Tester 3] → 审查打砖块
```

**适用场景**：同时审查多个已有项目

### 模式三：补救旁路（Bypass / Fallback）

当某个 Agent 超时或输出不完整时，跳过或由下一个 Agent 补位。

```
  Designer → [Artist 超时] → Architect 自行补全视觉设计 → Coder → ...
```

**适用场景**：子 Agent 超时（2min 限制）时的降级策略

## 实际实现：如何通过 OpenClaw API 协作

在 OpenClaw 平台上，多 Agent 协作通过以下机制实现：

### 1. sessions_spawn（子 Agent 调用）

```javascript
// 主 Agent 启动子 Agent
const child = await sessions_spawn({
  runtime: "subagent",
  mode: "run",
  task: "## Game Designer\n具体任务...",
  runTimeoutSeconds: 180
});

// 主 Agent yield，等待子 Agent 完成
// 子 Agent 完成后，结果以事件形式返回
```

### 2. 文件系统作为共享内存

```
Agent 不直接通信，而是通过文件系统传递产物：

  Designer → 写入 PRD.md → Artist 读取 PRD.md
  Artist  → 写入 视觉设计 → Architect 读取
  Architect → 写入 技术方案 → Coder 读取
  ...
```

这是最可靠的方式——不依赖 Agent 间的消息通道。

### 3. 主 Agent 编排

主 Agent（当前的你）负责：

1. **启动流水线**：按顺序 spawn 子 Agent
2. **扇出并行**：同时启动多个独立任务
3. **处理超时**：子 Agent 超时后降级策略
4. **合并结果**：汇总各 Agent 输出
5. **异常处理**：检查文件是否真实写入

## 项目的实际代码示例

### 创建新游戏的完整流水线（Python 伪代码）

```python
def create_game(idea):
    # Step 1: 策划
    prd = spawn_agent("game-designer", idea)
    save_file("PRD.md", prd)
    
    # Step 2-3: 并行执行
    vision = spawn_agent("pixel-artist", prd)
    arch = spawn_agent("game-architect", prd)
    wait_all([vision, arch])
    
    # Step 3.5: 如果 Artist 超时，Architect 补位
    if vision is None:
        arch += generate_textures(arch.specs)
    
    # Step 4: 编码
    code = spawn_agent("game-coder", prd + vision + arch)
    save_file("index.html", code)
    
    # Step 5: 测试
    issues = spawn_agent("game-tester", code)
    
    # Step 6: 如通过则部署
    if not issues.critical:
        spawn_agent("game-publisher", code)
    
    return "games/new-game/"
```

### 审查已有游戏的并行模式

```python
def review_all_games():
    games = ["dino-run", "space-shooter", "breakout"]
    
    # 并行审查所有游戏
    tasks = [spawn_agent("game-tester", g) for g in games]
    results = wait_all(tasks)
    
    # 合并所有审查结果
    return merge_reviews(results)
```

## 各 Agent 角色详解

### 🎯 Game Designer（策划官）
- **能力**：产品思维、游戏设计、文档撰写
- **依赖**：无（从零开始）
- **产物**：PRD.md（游戏策划文档）
- **成功标准**：文档包含核心机制、玩法说明、技术规格

### 🎨 Pixel Artist（原画师）
- **能力**：像素艺术、配色设计、Canvas 纹理代码
- **依赖**：PRD.md
- **产物**：视觉设计文档 + 纹理生成代码
- **已知限制**：容易超时，输出不完整 → 降级到 Game Coder 内嵌生成

### 🏗️ Game Architect（架构师）
- **能力**：系统设计、算法设计、性能优化
- **依赖**：PRD.md + 视觉设计
- **产物**：技术方案文档（数据结构、函数签名、算法伪代码）
- **已知限制**：容易超时，输出不完整 → 降级到 PRD 已含架构信息

### 💻 Game Coder（主程）
- **能力**：编码、Canvas/Three.js、问题解决
- **依赖**：PRD.md（最少依赖，必要时自给自足）
- **产物**：index.html（完整可运行游戏）
- **核心约束**：单文件、≤30KB（2D）/ ≤50KB（3D）、无外部依赖

### 🐛 Game Tester（测试员）
- **能力**：代码审查、Bug 发现、性能分析
- **依赖**：index.html
- **产物**：REVIEW.md（缺陷清单）
- **检查范围**：语法错误、功能缺失、性能问题、用户体验

### 🚀 Game Publisher（发布官）
- **能力**：文件整理、元数据管理、构建部署
- **依赖**：index.html（通过测试后）
- **产物**：games.json 更新 + GitHub Pages 部署

### 🕵️ Web Fixer（网页修复专家）
- **能力**：线上验证、URL 检查、浏览器渲染验证、SPA 修复
- **依赖**：GitHub Pages 部署完成后触发
- **检查方式**：`web_fetch` 检查 HTTP 状态 + 浏览器截屏看实际渲染
- **常见问题**：CDN 缓存延迟、路径大小写、SPA 404 fallback、CDN 被墙
- **修复流程**：发现问题 → 定位代码 → 修复 → 重新构建 → 推送 → 复查

## 常见故障模式与应对

| 故障 | 表现 | 应对策略 |
|------|------|----------|
| 子 Agent 超时 | 2min 后无输出 | 跳过该步骤，降级处理 |
| 输出不完整 | 只返回了摘要 | 检查是否有文件写入，无则降级 |
| 代码有 Bug | Tester 报告缺陷 | Tester 输出 Bug 清单 → 修复 |
| 游戏无法运行 | 加载空白 / 报错 | 读代码手动修复 |
| 部署失败 | GitHub push 失败 | 检查网络/密钥，切换方式 |

## 为什么这样设计

1. **文件系统作为共享内存**：比 Agent 间消息传递更可靠，不会因超时丢失进度
2. **容错优先**：每个环节都有降级方案，流水线不会因单个失败卡死
3. **最小依赖原则**：每个 Agent 只需 PRD 就能工作（其他产出可选）
4. **主 Agent 编排**：人类（你）或主 Agent 控制全局，子 Agent 只管干好自己的活
5. **主程万金油**：Game Coder 是唯一不可跳过的环节——它集成了 Designer 的创意、Artist 的视觉、Architect 的架构
